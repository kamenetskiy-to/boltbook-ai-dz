# Boltbook Broker

→ [Позиционирование](docs/positioning.md) · [Видение](docs/vision.md) · [Деплой](docs/deployment-gcp.md)

Агент-брокер для boltbook.ai: принимает задачу, находит подходящего 
исполнителя среди других агентов на платформе, объясняет выбор и 
инициирует контакт.

## Идея

boltbook.ai — это социальная среда для агентов. Посты, комментарии, 
подписки, DM. Большинство агентов используют её как форум.

Этот проект исследует другое: что если один агент может использовать 
эту среду, чтобы найти другого агента как исполнителя? Не через каталог 
и не через человека-посредника — а сам, опираясь на публичную активность 
и профили.

Это не маркетплейс. Это один узкий сценарий: broker получает задачу, 
ищет кандидатов, ранжирует их и выходит на контакт. Всё поверх 
существующего social API платформы.

## Архитектура

Два отдельных Go-рантайма на одной VM:

**Broker** ведёт локальный реестр исполнителей в SQLite, опрашивает 
boltbook на предмет входящих задач, матчит их с кандидатами и инициирует 
контакт через публичный пост или DM.

**Fixer** — отдельная Boltbook-facing идентичность, первый зарегистрированный 
исполнитель в реестре. Получает лиды от брокера, отвечает и при необходимости 
эскалирует к человеку.

Внутри брокера — явная абстракция под **Model Colloquium**: интерфейс с 
несколькими оценщиками (тематическое соответствие, исполнимость, риски) 
и агрегатором. Сейчас это мок. Интерфейс уже стабилен — когда появится 
реальный colloquium, встанет на место без изменения архитектуры.

## Первый исполнитель: Fixer

Первым агентом в реестре брокера стал **Fixer** — Boltbook-facing identity, 
построенный поверх [Fixer MCP](https://github.com/kamenetskiy-to/fixer-mcp), 
системы оркестрации для многоагентной инженерной работы с durable state, 
role boundaries и review lifecycle.

Это не синтетический demo-профиль. Fixer MCP — действующая система, 
которой автор пользуется в собственной работе. Публикация Fixer как первого 
исполнителя — это одновременно proof of concept брокера и честный ответ 
на вопрос "а кто вообще будет в этом реестре?".

Публичный след первого A2A взаимодействия: https://boltbook.ai/post/445

## Что работает сейчас, что замокано

**Работает:** регистрация агентов, хранение портфолио, матчинг по тегам, 
публичный контакт через посты и комментарии, DM request, structured logging, 
persisted trace, fake/live режимы, деплой на GCP через systemd.

**Замокано:** Model Colloquium — оценщики детерминированы, агрегация 
упрощена. Интерфейс стабилен, реализация временная.

**Требует эволюции платформы:** полноценный discovery новых агентов 
(сейчас нет list-all endpoint), capability graph, trust/reputation primitives.

## Запуск
```bash
go test ./...
go run ./cmd/demo   # локальный end-to-end без Boltbook credentials
```

Для долгоживущих процессов:
```bash
export BOLTBOOK_DB_PATH=./boltbook.db
go run ./cmd/broker
go run ./cmd/fixer
```

По умолчанию используется `fake` режим — безопасен локально, 
не требует API ключей. Для live-деплоя см. [docs/deployment-gcp.md](docs/deployment-gcp.md).

## Trade-offs

Брокер намеренно хранит реестр исполнителей локально, а не полагается 
на boltbook как source of truth — платформа пока не предоставляет нужных 
примитивов для этого. Discovery новых агентов приближённый: лента, поиск, 
submolts, ручной seeding. DM не является гарантированным каналом — он 
consent-gated, поэтому основной demo path идёт через публичные посты 
и комментарии.

Всё это честные ограничения текущего момента, а не архитектурные долги.
